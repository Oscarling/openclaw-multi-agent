# 本地优先 + 分段上云执行口径（v1）

日期：2026-03-26  
适用范围：`openclaw multi-agent` 主线推进期（加速模式）

## 1) 背景与目标

当前主线需要提速，逐步“做一点就上 GitHub”会带来明显等待成本。  
本口径采用：**本地阶段收口 -> 事件触发再分段上云（GitHub）**。

目标：
- 降低每次网络往返开销
- 保持可回滚、可审计
- 不改变既有协作门禁（PR + squash + 分支保护）

## 2) 触发式上云条件（不用时间节点）

满足任一条件即触发一次“分段上云”：

1. 一个主线事件收口
   例如：`R19+R20` 批处理完成，或某个 Gate 条件项关闭。
2. 一个风险项收口
   例如：安全修复、路由口径修复、恢复演练结论落盘。
3. 一个可回滚检查点形成
   例如：新增文档/脚本/角色边界后，已具备最小验证证据。

## 3) 执行步骤（本地阶段 -> 上云阶段）

### A. 本地阶段（连续推进）

1. 持续在同一个工作分支完成一批相关改动（不要频繁切分远端 PR）。
2. 每完成一个子事件就本地提交一次，保证可回退：
   - 代码或文档改动
   - 对应验证证据（日志/报告）
   - `BACKLOG.md` / `DECISIONS.md` / `验收清单.md` 对齐

### B. 上云阶段（一次性同步）

当触发条件满足时，统一执行：

```bash
git status -sb
git push origin <branch>
gh pr create --fill
```

说明：
- 仍按既有规则走 PR 合并，不直推 `main`。
- 若网络不稳，保持本地提交不丢失，恢复后继续 `git push`。

## 4) 边界与护栏

- secrets / token / `runtime/` 运行态不入库。
- 关键改动必须“本地提交 + 验证证据”后才可进入上云阶段。
- 每次上云应保持“同一主题”，避免一个 PR 混入多个无关改动。

## 5) 对主线节奏的影响

- 主线推进由“等待远端”改为“本地先跑通”。
- GitHub 从“实时同步工具”变为“阶段收口与协作审计工具”。
- 在不放松门禁的前提下，整体推进速度可明显提升。
